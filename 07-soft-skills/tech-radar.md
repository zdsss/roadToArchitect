# 技术雷达（Tech Radar）

技术雷达是 ThoughtWorks 提出的技术评估框架，帮助团队系统性地评估和管理技术选型。它把技术按照"应用阶段"分为四个环，把技术类型分为四个象限，形成一张可视化的技术地图。

---

## 1. 技术雷达的结构

### 1.1 四个象限（Quadrants）

| 象限 | 包含的内容 |
|------|-----------|
| 语言与框架（Languages & Frameworks） | Python, FastAPI, Spring Boot, React |
| 平台（Platforms） | Kubernetes, AWS, PostgreSQL, Kafka |
| 工具（Tools） | GitHub Actions, Prometheus, Grafana |
| 技术（Techniques） | 六边形架构, TDD, 金丝雀发布, ADR |

### 1.2 四个环（Rings）

| 环 | 名称 | 含义 | 行动建议 |
|----|------|------|---------|
| 最内环 | **Adopt（采用）** | 强烈推荐，已在生产中证明价值 | 新项目应该使用 |
| 第二环 | **Trial（试用）** | 值得尝试，有信心它对我们有价值 | 在低风险项目中试用 |
| 第三环 | **Assess（评估）** | 值得关注，但尚未足够了解 | 研究一下，做 spike 验证 |
| 最外环 | **Hold（暂缓）** | 谨慎使用或停止新项目使用 | 不在新项目中引入 |

```
        Adopt ←────────────────────────────────── Hold
           ↑                                          ↑
      已证明价值                               不推荐新项目使用
      (生产就绪)                              (有更好替代或风险高)
           │                                          │
           │    Trial           Assess                │
           │  (低风险试用)     (值得关注)              │
           └──────────────────────────────────────────┘
```

## 2. ShopFlow 技术雷达示例

以 ShopFlow 项目为例，建立一份小型团队的技术雷达：

### 2.1 Adopt（采用）— 新项目默认选择

**语言与框架：**
- Python 3.12：主要后端语言，团队熟悉，生态成熟
- FastAPI：异步高性能，自动 OpenAPI 文档，类型安全
- Pydantic v2：数据验证，与 FastAPI 深度集成

**平台：**
- PostgreSQL：核心事务数据，ACID，生态成熟
- Redis：缓存、会话、限流
- Docker + Docker Compose：本地开发环境标准化

**工具：**
- GitHub Actions：CI/CD 流水线
- pytest：Python 测试框架
- Prometheus + Grafana：指标监控

**技术：**
- REST API：外部 API 标准风格
- ADR：架构决策记录
- 分层架构：Controller/Service/Repository 分层

### 2.2 Trial（试用）— 在低风险项目中验证

**语言与框架：**
- Java 21 + Spring Boot 3：Sprint 10 开始引入微服务，需要验证

**平台：**
- RabbitMQ：Sprint 7 引入，还在验证生产稳定性
- Kubernetes：Sprint 13 引入，还在学习曲线上

**技术：**
- 六边形架构：Sprint 8 重构，验证实际可测试性收益
- OpenTelemetry：Sprint 12 引入，评估跨服务追踪效果

### 2.3 Assess（评估）— 值得研究，暂未采用

**平台：**
- Kafka：日消息量超过100万时的替代方案，现在开始研究
- MongoDB：商品目录 Schema 灵活性需求，评估是否比 PostgreSQL JSONB 更合适

**技术：**
- CQRS：读写分离，评估在订单历史查询场景中的适用性
- Service Mesh（Istio）：微服务规模到 20+ 服务时需要，现在开始了解

**工具：**
- Jaeger：分布式链路追踪，与 OpenTelemetry 配合，计划在 Sprint 12 评估

### 2.4 Hold（暂缓）— 不在新项目中引入

**语言与框架：**
- Flask：FastAPI 更现代，Flask 异步支持差，不在新项目中使用
- Django REST Framework：重，不适合微服务风格 API

**平台：**
- MySQL：团队选定 PostgreSQL，不引入第二个关系型数据库
- MongoDB（仅用于代替关系型）：如果只是因为"NoSQL 流行"而用，放在 Hold

**技术：**
- 分布式两阶段提交（2PC）：微服务中代价太高，用 Saga 替代
- 共享数据库（多服务共用一个 DB）：违反微服务原则，Hold

## 3. 如何评估一项技术

当遇到一项新技术（如：某同事提议引入 GraphQL），如何系统性地评估？

### 3.1 评估框架

```
维度1：解决的问题
  "它解决了什么具体问题？我们有这个问题吗？"
  → GraphQL 解决 over-fetching。我们有这个问题吗？前端团队有抱怨吗？

维度2：成熟度
  "它的生产成熟度如何？有大规模使用案例吗？"
  → GraphQL：GitHub、Twitter 在用。成熟。
  → 某新框架 v0.1：高风险，不适合生产。

维度3：团队成本
  "采用它的学习曲线、迁移成本是多少？"
  → GraphQL 需要学 schema、resolver、DataLoader，约2周上手。

维度4：生态与维护
  "社区活跃吗？有商业公司支持吗？"
  → 避免选择无维护者的技术。

维度5：替代方案
  "已有的技术能解决这个问题吗？"
  → REST + fields 参数能缓解 over-fetching。先试这个。
```

### 3.2 Spike（技术验证）

在决定引入前，做一个时间限制的验证：

```
Spike 计划：评估 GraphQL 是否适合 ShopFlow

目标：验证 GraphQL 能否降低移动端网络请求量
时间限制：3天
交付物：
  - 实现一个核心查询（商品列表）的 GraphQL 端点
  - 对比 REST vs GraphQL 的请求数据量
  - 测量实现复杂度（代码行数、学习时间）
  - 写一份 Assess/Trial/Adopt/Hold 的建议

通过标准：
  - 移动端数据传输量减少 > 30%
  - 实现复杂度在团队可接受范围内
```

## 4. 团队技术雷达的维护

### 4.1 更新频率

- **大型团队（ThoughtWorks 风格）：** 每6个月更新一次，发布正式版本
- **小型团队（ShopFlow）：** 每个 Sprint 结束时更新一次状态，无需正式流程

### 4.2 决策原则

| 原则 | 说明 |
|------|------|
| 保守引入 | 新技术先 Assess → Trial，不直接 Adopt |
| 主动 Hold | 发现问题立刻 Hold，不因为"已经用了"就容忍 |
| 有理有据 | 每次状态变更要记录理由（类似 ADR） |
| 全团队参与 | 技术雷达不是架构师一个人的，开发、运维都应参与评估 |


## Key Architect Takeaways

- **技术雷达让技术选型变成团队共识而非个人喜好**：当有人提议"我们用 X"时，不是看谁声音大，而是按框架评估后更新雷达
- **Hold 比 Adopt 更重要**：团队更容易引入新技术，但很少主动淘汰旧技术。技术雷达强迫你定期问："我们还需要这个吗？"
- **先 Assess 和 Trial，不要直接跳到 Adopt**：任何未经验证的技术都应该先小范围试用。从 Trial 到 Adopt 的路径是：Spike → 低风险项目实践 → 总结经验 → 团队评审通过
- **技术雷达是活文档，不是一次性报告**：每个 Sprint、每次技术决策，都是更新雷达的机会。只有持续维护的雷达才有价值
