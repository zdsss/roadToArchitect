# 沟通与影响力（Communication）

技术能力决定你能做什么，沟通能力决定你能做成什么。架构师不只是写代码的人，更是技术决策的推动者、跨团队协作的桥梁、技术与业务之间的翻译者。

---

## 1. 架构师的沟通挑战

| 挑战 | 场景 | 失败的表现 |
|------|------|-----------|
| 技术 → 业务 | 向 PM/CEO 解释为什么要做技术重构 | "我们需要解耦服务" → 对方听不懂，拒绝排期 |
| 复杂 → 简单 | 向新人解释微服务架构 | 画了30个方框的架构图 → 对方更困惑 |
| 推动共识 | 技术评审，团队有不同意见 | 会议结束，仍然没有决策 |
| 书面表达 | 写架构文档 | 文档没人读，或读完还是不懂 |

## 2. 向非技术人员解释技术

### 2.1 用业务语言，不用技术词汇

```
❌ 技术语言（非技术人员听不懂）：
"我们需要把单体应用拆分成微服务，引入 Kafka 做异步消息，
用 Redis 解决缓存穿透问题，并在 API Gateway 层实现熔断器。"

✅ 业务语言（关注影响，而非实现）：
"现在系统每次发布都要停机2小时。我们的目标是：
发布时用户无感知（零停机），并且每个功能团队可以独立发布，
不用等其他团队。预计需要3个月，之后发布频率可以从每月1次提升到每天多次。"
```

### 2.2 STAR 结构解释技术决策

**S**ituation（现状）→ **T**ask（任务/问题）→ **A**ction（我们做了什么）→ **R**esult（结果/影响）

```
"（S）我们的商品搜索页面平均加载时间是 8 秒，用户投诉增加了 30%。
 （T）需要把搜索响应时间降到 1 秒以内。
 （A）我们为商品数据加了缓存层，热门商品的查询直接从内存读取。
 （R）搜索响应时间从 8 秒降到了 0.3 秒，用户投诉量下降了 85%。"
```

### 2.3 技术债的业务解释

```
❌ "我们需要重构代码，技术债很重。"
   → PM：重构有什么用户价值？我有新功能要排期。

✅ "我们的发布速度在下降。三个月前，一个新功能平均2天开发完。
   现在同样的功能需要5天，因为我们每次改一处，都需要手动测试
   三个地方有没有被影响。这个数字还在继续变差。
   如果投入4周做架构整理，预计发布速度恢复到2天，
   并且每个功能的 Bug 率会下降 50%。"
```

## 3. 架构文档：C4 模型

C4 模型（Context, Containers, Components, Code）是一种分层的架构可视化方法，用不同抽象级别面向不同受众：

| 层级 | 面向受众 | 内容 | 详细程度 |
|------|---------|------|---------|
| Level 1: System Context | 所有人（含非技术） | 系统与外部用户/系统的关系 | 极简 |
| Level 2: Container | 技术管理者、架构师 | 应用程序、数据库、API | 中等 |
| Level 3: Component | 开发者 | 服务内部的主要模块 | 详细 |
| Level 4: Code | 开发者 | 类图、函数 | 极详细（通常自动生成） |

**Level 1 示例（ShopFlow System Context）：**
```
外部用户（买家）
    │
    ▼
[ShopFlow 电商平台]
    │                    │
    ▼                    ▼
[支付网关（Stripe）]  [邮件服务（SendGrid）]
```

**Level 2 示例（Containers）：**
```
[浏览器/移动App]
    │ HTTPS
    ▼
[API Gateway（Nginx）]
    │        │        │
    ▼        ▼        ▼
[user-svc] [product-svc] [order-svc]
    │            │            │
    ▼            ▼            ▼
[users_db]  [products_db]  [orders_db]
(PostgreSQL) (PostgreSQL)  (PostgreSQL)
                │
                ▼
            [Redis 缓存]
```

### 实用建议：用文本即代码（Diagram as Code）

```python
# 用 Mermaid（在 Markdown 里直接渲染）
# 放在 docs/architecture.md 里，随代码一起版本化

"""
graph TB
    User([买家])
    Admin([管理员])

    subgraph ShopFlow
        Gateway[API Gateway]
        UserSvc[用户服务]
        ProductSvc[商品服务]
        OrderSvc[订单服务]
    end

    User --> Gateway
    Admin --> Gateway
    Gateway --> UserSvc
    Gateway --> ProductSvc
    Gateway --> OrderSvc
"""
```

## 4. 主持技术评审

技术评审（Tech Review）的目标是：**在一小时内，让5个人对一个架构决策达成共识**。

### 4.1 评审会议结构

```
会议前（必须做）：
  发送预阅读材料（设计文档草稿），给大家至少24小时阅读
  明确本次评审的决策范围（"我们要决定：用什么消息队列"）

会议中（60分钟）：
  0-10分钟：作者快速介绍背景和方案（不读文档，讲重点）
  10-40分钟：开放讨论，逐一考虑方案和权衡
  40-55分钟：明确决策（不是"讨论很好"，而是"我们决定做X"）
  55-60分钟：记录下一步行动项（谁、做什么、什么时候）

会议后（必须做）：
  24小时内发送会议纪要：决策 + 理由 + 行动项
```

### 4.2 推动共识的技巧

```
当讨论陷入僵局时：

技巧1：时间限制决策
  "我们还有15分钟。如果没有达成共识，我们默认选方案A，
   如果有人反对，请在接下来5分钟内提出具体的反驳。"

技巧2：分离事实与观点
  "你说 Kafka 比 RabbitMQ 好——这是事实吗？在什么维度上？
   在我们的规模（日消息量5万）上，这个差异有多大？"

技巧3：明确关键约束
  "我们最不能妥协的是哪一点？性能、一致性还是开发成本？
   如果是开发成本，那可能 RabbitMQ 是更好的选择。"

技巧4：共识确认
  "我理解的是：我们决定用 RabbitMQ，理由是当前规模不需要 Kafka，
   当日消息量超过100万时重新评估。大家同意吗？"
```

## 5. 书面沟通：写让人愿意读的文档

### 5.1 架构文档的结构

```markdown
# [系统/功能名称] 设计文档

**状态：** 草稿 / 评审中 / 已批准
**作者：** [姓名]
**评审者：** [相关人员]
**最后更新：** YYYY-MM-DD

## TL;DR（3句话总结）
[不愿意读完整文档的人，读这里就够了]

## 背景
[为什么需要做这个？现状是什么？]

## 目标
[这个设计要解决什么问题？成功的标准是什么？]

## 非目标（Out of Scope）
[明确说明什么不在这次设计范围内，防止范围蔓延]

## 方案
[推荐方案的详细描述]

## 备选方案
[考虑过但未选的方案，及简短说明]

## 权衡分析
[参考 trade-off-analysis.md 的格式]

## 实施计划
[分阶段，每阶段的交付物]

## 未解问题（Open Questions）
[还没有答案的问题，及由谁、什么时候回答]
```

### 5.2 写作原则

| 原则 | 好的示例 | 差的示例 |
|------|---------|---------|
| 结论先行 | "我们推荐使用 Redis 缓存，理由如下..." | "缓存有很多选择，Redis 是一个，Memcached 是另一个...（5段后才说推荐谁）" |
| 数据支撑 | "搜索响应从 8s → 0.3s，降低 96%" | "性能提升显著" |
| 明确读者 | 标注"本节面向技术评审者" | 混合面向所有人 |
| 避免废话 | "我们面临 X 问题" | "在当今快速发展的互联网时代，系统性能日益重要..." |

## Key Architect Takeaways

- **沟通的第一原则是了解受众**：向 CEO 解释用业务影响，向新工程师解释用比喻和类比，向同级架构师直接讨论技术细节——三种场景，三种语言
- **C4 模型解决"一张图不够用"的问题**：从系统上下文到代码细节，用四个层次满足不同受众的需求，不要用一张图试图表达所有信息
- **TL;DR 是对读者最大的尊重**：大多数人不会读完你的文档。在最前面放 3 句话的总结，让人在 30 秒内理解你的核心观点
- **推动决策比完善方案更重要**：架构评审常见的失败是"讨论了3次，还没有决策"。会议结束时必须有明确的决策，哪怕是"我们下周再讨论一次"也是一个决策
