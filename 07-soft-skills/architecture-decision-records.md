# 架构决策记录（Architecture Decision Records）

架构决策的最大问题不是做错了决策，而是**忘记了为什么这样做**。六个月后，没有人记得当时为什么选了 RabbitMQ 而不是 Kafka，也没有人知道那个"奇怪的缓存设计"背后有什么考量。ADR（架构决策记录）解决的就是这个问题。

---

## 1. 什么是 ADR

ADR 是一份简短的文档，记录：
- **背景**：做这个决策时面临什么问题，有哪些约束
- **决策**：我们选择了什么
- **理由**：为什么选这个，为什么不选其他
- **后果**：这个决策带来什么，包括好的和坏的

ADR 不是设计文档（不描述系统怎么工作），不是需求文档（不描述要做什么），而是**决策的历史记录**。

---

## 2. 为什么要写 ADR

| 场景 | 没有 ADR | 有 ADR |
|------|---------|--------|
| 新人入职 | "为什么用 dict 不用 list？""不知道，一直这样的" | 直接读 ADR-001，5分钟理解决策背景 |
| 重新评估旧决策 | 不知道当时的约束，无法判断是否仍然适用 | ADR 里有"迁移信号"，知道什么时候应该重新考虑 |
| 技术评审 | 口头解释，三个月后没人记得 | 文档存在，可随时查阅 |
| 架构演化 | 推翻旧决策，不知道会影响什么 | 新 ADR 引用旧 ADR，建立决策链 |

---

## 3. ADR 格式

ADR 应该简短——1-2页，不是论文。标准格式：

```markdown
# ADR-NNN: [决策的标题（动词短语）]

**状态：** 草稿 / 已批准 / 已废弃 / 已被 ADR-XXX 取代
**日期：** YYYY-MM-DD
**决策者：** [姓名或团队]

## 背景（Context）

[2-5句话描述：做这个决策时的情况，面临的问题，相关约束]

## 决策（Decision）

我们将 [做什么]。

## 理由（Rationale）

[为什么选这个方案，为什么不选其他方案]

## 后果（Consequences）

**正面：**
- ...

**负面/代价：**
- ...（及应对方式）

## 迁移信号（When to Revisit）

[什么情况下应该重新考虑这个决策]
```

## 4. ShopFlow 项目的三个 ADR 示例

### ADR-001：选择 Python dict 作为商品内存存储

```markdown
# ADR-001: 选择 Python dict（而非 list）存储商品数据

**状态：** 已批准
**日期：** 2026-03-26
**决策者：** ShopFlow 开发团队

## 背景

Sprint 1 需要在内存中存储商品数据，支持按 ID 增删查改。
当前商品数量预计 < 10,000 个。主要操作：按 ID 查询（高频），按名称搜索（低频）。

## 决策

使用 Python `dict`，key 为商品 ID（整数）。

## 理由

按 ID 查询是最频繁的操作（GET /products/{id} 占查询的 ~90%）。
- dict：O(1) 查找，直接 `store[id]`
- list：O(n) 查找，需要遍历所有元素

按名称搜索（O(n)）频率低，即使是 10,000 个商品，遍历时间 < 1ms，可接受。

被否决的方案：
- list：按 ID 查找 O(n)，在商品数量增长时性能线性下降

## 后果

正面：
- 按 ID 查询 O(1)，性能不随商品数量增长而下降

负面：
- 按名称搜索仍是 O(n)
  → 可接受，数量级（<10,000）下影响可忽略
  → 迁移信号触发时引入搜索索引

## 迁移信号

- 商品数量超过 100 万，且按名称搜索成为性能瓶颈 → 引入 Elasticsearch
```

### ADR-002：选择 REST 而非 GraphQL

```markdown
# ADR-002: 选择 REST API 风格（而非 GraphQL）

**状态：** 已批准
**日期：** 2026-03-26
**决策者：** ShopFlow 开发团队

## 背景

Sprint 2 需要为 ShopFlow 设计 HTTP API，暴露给 Web 前端和移动端。
团队规模：3人。前端需求：商品浏览、购物车、订单管理，字段需求相对固定。

## 决策

选择 REST API，使用 FastAPI 实现，遵循 RESTful 设计原则。

## 理由

**选择 REST 的理由：**
1. 团队成员对 REST 熟悉，GraphQL 需要 1-2 周学习成本
2. 前端数据需求不复杂，over-fetching 问题不显著
3. HTTP 缓存（CDN）对 REST GET 请求天然支持
4. 工具链成熟（Postman、curl、浏览器 DevTools 直接可用）
5. FastAPI 自动生成 OpenAPI 文档，文档成本接近零

**否决 GraphQL 的理由：**
- 解决的问题（精确字段选择）在当前阶段不是痛点
- 额外引入 schema 定义、resolver、DataLoader 的复杂度

**否决 gRPC 的理由：**
- 面向浏览器的 API，gRPC 需要额外的 Web 代理
- 团队没有 Protobuf 经验

## 后果

正面：
- 学习成本低，可以立即开始开发
- 天然支持 HTTP 缓存

负面：
- 移动端可能存在 over-fetching（返回了前端不需要的字段）
  → 短期可接受；如果成为问题，可以在特定端点添加 `fields` 参数
- 没有 GraphQL 的类型自查能力

## 迁移信号

- 移动端团队频繁反映带宽浪费、字段不匹配 → 考虑为移动端单独提供 GraphQL 端点
- 前端需要复杂数据组合查询 → 考虑 BFF（Backend for Frontend）模式
```

### ADR-003：选择 RabbitMQ 而非 Kafka（Sprint 7）

```markdown
# ADR-003: 选择 RabbitMQ 实现异步消息（而非 Kafka）

**状态：** 已批准
**日期：** 2026-03-26
**决策者：** ShopFlow 开发团队

## 背景

Sprint 7 需要将"下单后发邮件"和"下单后扣库存"改为异步处理。
当前消息量：日订单量约 10,000，日消息量 < 50,000。
团队消息队列经验：有 RabbitMQ 经验，无 Kafka 生产经验。

## 决策

使用 RabbitMQ + Celery 实现异步消息处理。

## 理由

**选择 RabbitMQ 的理由：**
1. 消息量级（< 5万/天）远未达到需要 Kafka 的规模（百万/天）
2. RabbitMQ 的 Topic Exchange 满足"发邮件"和"扣库存"的路由需求
3. 团队有 RabbitMQ 使用经验，Kafka 需要学习 ZooKeeper/KRaft
4. Kafka 的运维（副本、分区管理）复杂度超出当前团队能力
5. 不需要消息回放（历史订单消息不需要重放）

**否决 Kafka 的理由：**
- 运维 ZooKeeper 集群成本高
- 消息量级不需要 Kafka 的高吞吐
- 团队学习成本：Kafka 生产部署需要 2-3 周实践

## 后果

正面：
- 快速实现异步处理，团队上手快
- RabbitMQ 的可视化管理界面方便调试

负面：
- 消息消费后删除，无法回放历史消息
  → 当前业务不需要回放，可接受
- 单节点 RabbitMQ 有性能上限（~5万/秒）
  → 当前规模（50/秒）距上限极远

## 迁移信号

- 日消息量超过 100 万 → 评估迁移到 Kafka
- 需要消息回放（如：审计、事件溯源）→ 必须迁移到 Kafka
- 需要精确的消息顺序保证（跨消费者）→ 需要 Kafka 的分区顺序
```

## 5. ADR 的存储和管理

```
docs/
└── decisions/
    ├── ADR-template.md
    ├── ADR-001-storage-data-structure.md
    ├── ADR-002-api-style-rest-vs-graphql.md
    ├── ADR-003-message-queue-rabbitmq.md
    └── ADR-004-...

命名规范：
ADR-{NNN}-{kebab-case-标题}.md
NNN：三位数字（001, 002, ...），按时间顺序编号

状态流转：
草稿 → 已批准 → 已废弃（原决策不再适用）
                → 已被 ADR-XXX 取代（有新决策替代）
```

**废弃示例：**
```markdown
# ADR-001: 选择 Python dict 存储商品数据

**状态：** 已被 ADR-008 取代
**取代理由：** 商品数量超过 100 万，dict 无法支持高效的全文搜索

---
（原内容保留，不删除）
```

> **不要删除旧 ADR**：即使决策已经废弃，也要保留原文档（只改状态）。这份历史记录本身就有价值——它告诉未来的工程师：我们曾经这样做，因为当时的约束是 X，后来因为 Y 才改变。

## 6. 什么时候写 ADR

| 触发条件 | 是否写 ADR |
|---------|-----------|
| 选择数据库（PostgreSQL vs MongoDB） | ✅ 写 |
| 选择架构风格（单体 vs 微服务） | ✅ 写 |
| 选择通信方式（REST vs gRPC vs 消息队列） | ✅ 写 |
| 确定认证方案（JWT vs Session） | ✅ 写 |
| 第三方服务选型（邮件服务、支付网关） | ✅ 写 |
| 选择变量命名规范 | ❌ 不写（粒度太细） |
| 决定函数放在哪个文件里 | ❌ 不写 |
| 修复一个 Bug | ❌ 不写（除非 Bug 揭示了架构问题） |

---

## Key Architect Takeaways

- **ADR 的价值在六个月后才真正体现**：今天写 ADR 看似多余，但半年后当你问"为什么这么设计"时，ADR 是唯一的答案来源
- **简短胜过完整**：ADR 不是设计文档，不需要覆盖所有细节。一份能在5分钟内读完的 ADR 比一份需要1小时的设计文档更有价值
- **状态很重要**：过时的 ADR 比没有 ADR 更危险（误导读者）。每次相关决策变化时，先更新旧 ADR 的状态，再写新 ADR
- **ADR 是团队共识的载体**：写 ADR 的过程（讨论背景、列举方案、分析权衡）本身就比结论更有价值——它强迫团队把隐性假设变成显性表达
